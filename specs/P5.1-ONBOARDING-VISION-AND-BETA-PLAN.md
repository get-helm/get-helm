**Author:** product-manager (CPO)
**Date:** 2026-06-15
**Trigger:** First beta test stopped early — install dead-ended, UX was confusing, info was wrong.
**Status:** Decision required (the install-path commitment is Level 4-5).

---

## 0. The one thing that broke the beta (root cause)

**Our entire Phase 2 is built on a false premise.** The P5.1 spec (line 48) says:

> "Claude Desktop Cowork — Cowork has filesystem access to actually install HELM."

**This is false, and it is the reason the beta died.** Researched and confirmed 2026-06-15:

- **Claude Cowork runs in an isolated Linux VM sandbox** (Apple Virtualization Framework). The hostname/`/sessions/` path/`Linux aarch64` the tester saw is the Cowork sandbox signature.
- It **cannot touch the host Mac filesystem**, cannot install Homebrew/Node on the Mac, cannot start a 24/7 process on the Mac. Anthropic's own engineering states: *"Cowork cannot install software on the host machine."*
- So when our prompt said "paste into Cowork, it installs HELM on your Mac," **every user will dead-end here.** It is not a copy bug. It is an architectural impossibility.

**The only Claude surfaces that can install local software + start a persistent service on the user's own Mac:**

| Surface | Can install on the Mac? | Terminal required? |
|---|---|---|
| Claude **Code CLI** | ✅ Yes (same local engine) | Yes — user types a command to launch `claude` |
| Claude Desktop → **Code tab → "Local" environment** | ✅ Yes (same local engine) | No *typed commands during the session*, but GUI app install + sign-in required |
| Claude Desktop → **Cowork tab** | ❌ No — sandbox | n/a (impossible) |
| Claude.ai web / cloud agent | ❌ No — sandbox | n/a (impossible) |

🔬 **Unverified and must be tested before we trust it:** whether Claude Desktop's **Code/Local** mode will run a *full unattended* Homebrew + clone + npm + background-service install for a non-technical user without choking on per-command approval prompts. The product design supports it; the end-to-end non-technical flow is unproven.

### Secondary failures (real, but downstream of the above)

1. **The README is a terminal-first quickstart with no link to the landing page.** It leads with `curl … install.sh | bash`. When a user (or a chat AI) is handed the README, it drags them straight into terminal work — exactly failure (a)+(b). The README never mentions get-helm.github.io.
2. **{{USER_JERRY}}'s first email pointed at the raw repo, not the landing page.** (The landing page itself is good — see below.)
3. **The richer install prompt was never published.** Two local versions exist; the *thinner* one (`helm-cowork-install-prompt.md`) is what's live. The richer one (`helm-cowork-prompt.md`) with config hydration, a Discord connection test, auto-start, and second-brain setup was authored but never shipped.
4. **Stale copy** in `public-docs/` ("30 minutes" / "Mac Mini-only" vs. the page's "45–60 min" + Windows-friendly).

### What actually worked (keep it)

- **The landing page (https://get-helm.github.io/) is correct and spec-compliant.** It has the Phase-0 CTA ("📋 Copy setup prompt → open Claude.ai"), hands out the *correct* Phase-1 pre-install prompt, and includes the "skip to install" path. The tester's experience confirmed the conversational Phase-1 flow was easy and good.
- The conversational AI-guided prep (Phase 1) is the right model and the tester sailed through it.

## 1. The complete onboarding vision (corrected)

The experience, end to end, for "my friend recommended this":

```
ENTRY        Friend sends ONE link → get-helm.github.io (never the repo)
PHASE 0      Landing page. One button: copy prompt + open Claude.ai. Zero commitment.
PHASE 1      Claude.ai (free, daily machine). Conversational guide:
             hardware check → optional wipe → Claude subscription →
             install Claude Desktop on the target machine →
             *** open the CODE tab, set environment to LOCAL *** (NOT Cowork)
PHASE 2      Claude Desktop → Code/Local on the always-on machine.
             AI runs the install, creates the Discord bot (click-by-click),
             hydrates config, runs a connection test, sets auto-start,
             confirms HELM's first Discord message landed.
PHASE 3      Discord. 3 preference taps → live channel tour → first value
             ("what's on my calendar?") → connectors → email backup.
LIVE         HELM is running 24/7. Day-3 and Day-7 nudges collect the rest.
```

**The non-negotiable correction:** Phase 1 and Phase 2 must direct the user to **Claude Desktop's Code tab set to "Local"** (or, as a fallback, the Claude Code CLI). The word "Cowork" must be removed from every install instruction. This single change is the difference between "works" and "impossible."

**The honesty correction:** We must stop promising "you will never touch a terminal." The truthful promise is: *"You won't have to understand or type terminal commands — the AI runs them for you. You will install one app (Claude Desktop) and sign in, the same as installing any Mac app."* If we choose the CLI fallback, one launch command is unavoidable and we should present it as a single copy-paste, framed plainly.

## 3. Order of operations (to reach a trustworthy beta)

1. **Decide the install path** (Decision A below). Everything else depends on it.
2. **Rewrite the README** (C4) → single clear CTA to get-helm.github.io; remove the curl\|bash quickstart from the top.
3. **Fix Phase-1 + Phase-2 prompts** (C2, C3): replace every "Cowork" with "Code tab → Local"; merge the rich Phase-2 prompt (hydration, connection test, auto-start, optional second-brain) and publish ONE canonical version.
4. **Soften the "no terminal" promise** to the honest version everywhere it appears.
5. **Verify today's shipped pieces actually fire** (C8 init, C11 tour, C12 lifeline) — read-back + a real spawn test, not "marked done."
6. **Publish** the corrected landing page + repo + prompts. Re-run PII scrub (C14).
7. **WE test it end-to-end on a real, near-clean Mac** before any human beta. This is the gate that has been skipped every time. No exceptions.
8. **Then** hand it to one friendly beta tester, watching live, with a feedback capture path.

## 5. Decisions required from {{USER_JERRY}}

**Decision A — Install path for beta (Level 4-5; changes the core UX promise):**
- **A1 — Claude Desktop → Code tab → Local.** Closest to the "no-terminal" vision; user installs one GUI app + signs in, AI does the rest. 🔬 *Risk:* unattended non-technical install in Code/Local is unverified — we must test it ourselves first, and it may hit per-command approval prompts.
- **A2 — Claude Code CLI.** Most reliable, proven to do local installs. *Cost:* user runs ~1–2 copy-paste terminal commands. Less magical, but it works today.
- **A3 — Hybrid (recommended):** Lead with A1 for the magic; if Code/Local stalls on approvals during our own test, fall back to A2's single command. Decide which is primary only after our own clean-Mac test.

**Recommendation: A3.** We don't actually know A1 works for a non-technical user until we run it ourselves. Committing to A1 sight-unseen is how we got here. Test both, ship the one that survives.

**Decision B — Second brain in beta or after?** The rich prompt includes QMD/second-brain setup (steps J–K). It adds install time and failure surface. Recommend: **post-beta** — get the core loop trustworthy first.

## 5b. Gaps found on the 2026-06-15 re-audit (NOT in the original checklist)

These three are new — they passed every prior review because they only surface on a real install or first run. Each is verified against source, not assumed.

**G-A — The $20 Pro promise may not survive a 24/7 bot (verified concern).**
P5.1 promises "Claude Pro ($20/month) — that's enough for HELM" (spec lines 77, 275). A bot running health checks every 6h, daily briefings, PM sweeps, and on-demand work will burn weekly usage fast. Web-confirmed: the Claude **Code tab itself requires Pro/Max**, and heavy agentic use hits weekly limits. We have never measured HELM's real weekly token burn against a fresh Pro cap. Risk: beta user's HELM goes quiet mid-week and they think it's broken. *Action:* either set the honest expectation ("Max recommended for always-on") or measure burn before promising Pro.

**G-C — Credential-storage contradiction (verified).**
The spec narration says "[SAVE token to HELM Vault — never stored in plain text]" (spec line 512) and "Bot token → HELM Vault only" (line 950). But the actual install writes the Discord token to `marvin-bot/.env` in plaintext (setup-headless.sh line 98; bot.js reads `process.env.DISCORD_BOT_TOKEN`). A new user has **no 1Password / PAP Vault** — that's {{USER_JERRY}}-specific infra. Good news: the core loop does NOT need the vault (.env works). Bad news: the onboarding script promises something false. *Action:* fix the wording to match reality (.env with correct file perms), or genuinely encrypt. Do not ship a false security promise.

**G-E — First daily briefing will be empty, contradicting the promise (verified).**
Onboarding tells the user "your daily briefing starts tomorrow at 8am" showing "calendar events, email summary, and weather" (spec lines 770, 797, 835). But connectors (Gmail/Calendar) are **deferred to post-beta** (spec line 978) — they're claude.ai MCP connectors tied to an account, not auto-provisioned. So a beta user's first briefing fires with no calendar/email data — exactly the "it's broken" first impression we're trying to avoid. *Action:* either gate the briefing promise behind connector setup, or make the first briefing gracefully say "connect your calendar to fill this in" instead of showing an empty shell.

**Verification gap (must check, not assume):**
- **G-D — Onboarding resume:** spec promises typing "onboarding" resumes from the last `ONBOARDING_STEP` (spec lines 930-934). Not yet verified that bot.js actually persists and resumes this. If a user's install is interrupted, do they restart from zero? Untested.

## 6. The meta-lesson

Every one of these failures traces to the same root: **we declared things done and packaged them without once installing from the published artifacts on a target machine.** Seven on-machine review passes can't catch what only a clean install reveals. The checklist above is worthless without line "The gate." That gate is the single highest-value change to how we work toward beta.
