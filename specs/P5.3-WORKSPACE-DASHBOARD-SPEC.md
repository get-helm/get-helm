# P5.3: Workspace Dashboard & Visibility — Complete Specification

**Status:** Design (awaiting approval)

**Option:** B now (pinned status per workspace) → C later (mission-control hub)

## Short-Term Solution (2 hours): Per-Workspace Status Pinned Message

**How it works:**

Every workspace has a pinned status message showing:
- Current phase
- What's in progress
- Last update timestamp
- Next steps

**Location:** Pinned in each workspace channel  
**Updated:** By PM sweep (every 30 min) + whenever workspace state changes

**Example:**

```
📊 [Workspace] Options-Monitoring — Status

Phase: B — Build & Validate
━━━━━━━━━━━━━━━━━━━━━━━━━━━
Current Loop: 2/5
Status: Testing margin alerts
Last update: Today 2:15 PM
Next: Debug false-positive alerts

Progress:
  ✅ Loop 1 — Fetch positions from brokerages
  🔄 Loop 2 — Calculate margin ← you are here
  ⏳ Loop 3 — Alert when margin > 80%
  ⏳ Loop 4 — Dashboard showing all accounts
  ⏳ Loop 5 — Mobile notification

Questions? Ask in this channel or @HELM help [topic]
```

**Implementation:**

1. **Scaffolder:** Auto-creates pinned message when workspace is created
2. **PM sweep:** Updates the pinned message every sweep (new phase, new loop, status change)
3. **Workspace agent:** Also updates when it posts a DELIVER (new phase reached, loop complete)

**Template (scaffolder writes this initially):**

```
📊 [WORKSPACE_NAME] — Status

Phase: [A|B|C|D]
Current: [what's happening now]
Last update: [timestamp]
Next: [what comes next]

Progress: [if in Phase B, show loop progress]

Questions? Ask in this channel or @HELM help [topic]
```

**Update logic (PM writes this on every sweep):**
```
For each workspace:
  1. Read WORKSPACE-PHASE.md (current phase + loop)
  2. Read workspace channel state (last message timestamp)
  3. Format status message
  4. Call discord-update-pinned.sh (update pinned message)
```

## Long-Term Solution (8 hours): Mission-Control Workspace Hub

**Design intent:** mission-control becomes your HELM dashboard — shows all 6–10 workspaces at a glance.

**How it works:**

mission-control channel shows:
- All active workspaces (cards showing phase + status)
- Quick actions (view workspace, pause automation, check latest proposal)
- Phase breakdown (X in Phase A, Y in Phase B, Z done)
- Recent activity feed

**Example Layout:**

```

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
📊 Active Automations (6)

[options-monitoring]
Phase: B · Loop 2/5 · Last update: today 2:15 PM
Status: Testing margin alerts
→ View | Pause | Archive

[daily-digest]  
Phase: C · Optimized · Last update: yesterday 10 AM
Status: Running successfully
→ View | Check settings

[japan-trip]
Phase: D · Live · Last update: 3 days ago
Status: Idle, waiting for input
→ View | Archive

[etf-dashboard]
Phase: C · Optimizing · Last update: today 9 AM
Status: Refining data pipeline
→ View | Check logs

   @HELM [describe what you want automated]
```

**Implementation:**

1. **mission-control template:** Created when HELM first runs (scaffolder)
2. **Status update job:** PM updates mission-control dashboard every sweep
3. **Quick actions:** Buttons link to individual workspaces + archived list
4. **Activity feed:** Shows last 10 actions across all workspaces

**What triggers updates:**
- Workspace phase changes → refresh dashboard
- Workspace loop completes → add to activity feed
- Workspace goes stalled (no activity 7+ days) → highlight in red
- New workspace created → add to dashboard
- Workspace archived/deleted → remove from active list

## Notes

**Why not build the hub now?** 2-hour solution gives you workspace visibility immediately. The hub is nice-to-have, not essential for HELM to work.

**Why mission-control instead of a web dashboard?** Stays in Discord. No new infrastructure. Consistent with HELM's philosophy (in-Discord automation, not {{USER_DOMAIN}}).

**What happens to per-workspace pinned messages when we build the hub?** They stay. Both exist. The hub is for overview; the workspace message is for detail.
